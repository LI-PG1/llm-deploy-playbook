# 08 · 代码适配

模型已能加载并对外服务，但加载与推理代码中常有若干细节需要适配——路径、设备分配、远程代码、量化加载、tokenizer 与多模态等。本篇逐一处理这些常见问题。

## 加载路径选择

加载模型时有四种典型路径，各自的使用场景和注意事项：

| 路径             | 使用场景         | 代码                                                | 常见问题                   |
| :------------- | :----------- | :------------------------------------------------ | :--------------------- |
| **HF Hub（在线）** | 首次部署、测试阶段    | `from_pretrained('repo/model')`                   | 网络不稳定、gated 模型需 token  |
| **本地目录（完整权重）** | 已下载到平台，生产环境  | `from_pretrained('/path/to/model')`               | 路径含特殊字符、缺少 config.json |
| **仅权重（需手动加载）** | 非标准框架/特殊格式   | `safetensors.torch.load_file()` + 自行构建模型          | 无自动架构识别                |
| **微调后检查点**     | LoRA/全量微调后推理 | `from_pretrained('./output_dir/checkpoint-1000')` | 缺少 tokenizer 文件        |

### 加载路径的通用判断逻辑

```python
# 1. 确认路径下有什么
import os
files = os.listdir('/path/to/model')
# 应该有：config.json + model.safetensors / pytorch_model.bin

# 2. 判断走哪个路径
if os.path.exists('config.json'):
    # 完整模型目录 → 标准加载
    model = AutoModel.from_pretrained('/path/to/model')
elif os.path.exists('adapter_config.json'):
    # LoRA 适配器
    from peft import PeftModel
    model = PeftModel.from_pretrained(base_model, '/path/to/adapter')
else:
    # 仅权重 → 终极方案
    pass
```

### config.json 常见陷阱

不同模型系列的 config 字段命名差异：

| 概念      | 常见别名                                         | 影响            |
| :------ | :------------------------------------------- | :------------ |
| 隐藏层维度   | `hidden_size`, `d_model`, `dim`, `n_embd`    | 模型定义失败        |
| 层数      | `num_hidden_layers`, `num_layers`, `n_layer` | 权重无法匹配        |
| 注意力头数   | `num_attention_heads`, `num_heads`, `n_head` | 推理结果错误        |
| 词汇表大小   | `vocab_size`（唯一标准名）                          | tokenizer 不匹配 |
| MoE 专家数 | `num_local_experts`, `num_experts`           | MoE 路由失效      |
| 专家激活数   | `num_experts_per_tok`, `top_k`               | 推理结果差异大       |

```python
# 查看 config.json 中的字段差异
import json
with open('config.json') as f:
    config = json.load(f)

hidden_size = config.get('hidden_size') or config.get('d_model') or config.get('dim')
print(f"Detected hidden_size: {hidden_size}")
```

> 🔧 DeepSeek-R1-Distill-Qwen-14B 的 config.json 同时存在 `num_hidden_layers` 和 `num_layers` 两个字段，值为 14 且一致。最初用 `from_pretrained` 自动加载时没问题，但手动构建模型类（Ultimate Solution）时用了错误的字段名，导致层数读成 0。

## device\_map 选择

不同规模模型的 device\_map 策略：

```python
# ≤3B 模型
model.cuda()                     # 简单直接，零依赖

# >3B 模型（＞7B MoE 必须 device_map="auto"）
model = AutoModel.from_pretrained(
    'model-path',
    device_map='auto',           # 自动分配各层到 GPU/CPU
    torch_dtype=torch.bfloat16
)
```

> `device_map='auto'` 依赖 `accelerate`，镜像里可能没有。小模型用 `model.cuda()` 更稳。

### device\_map 的 3 种模式

| 模式                                  | 效果                        | 适用场景      |
| :---------------------------------- | :------------------------ | :-------- |
| `'auto'`                            | accelerate 按层大小分配 GPU/CPU | 通用，MoE 必选 |
| `{0: '20GB', 'cpu': '60GB'}`        | 限制 GPU 显存分配量              | 单卡显存有限时   |
| `{'layer.0': 0, 'layer.1': 1, ...}` | 手动逐层指定                    | 调试/特殊需求   |

> 🔧 MoE 模型不用 device\_map 的后果：所有权重默认放 CPU，deepcopy 遍历数十 B 权重 → 永久卡死。
>
> 🔧 某 35B 模型 FP8 反量化到 BF16 后，40GB 单卡规格 OOM 54%，最终用 `max_memory={0: '20GB', 'cpu': '60GB'}` 才加载成功。

## trust\_remote\_code

```python
# 先看 config.json 的 architectures 字段
# 标准架构（Phi3/Llama/Qwen2/Mistral）：不设 trust_remote_code，用 transformers 内置实现
# 非标准架构（MiniCPM/Qwen3 MoE/Keye-VL）：必须设 trust_remote_code=True
```

**什么时候必须设**：

- architectures 字段包含自定义命名（如 `Qwen3ForCausalLM`、`MiniCPMForCausalLM`）
- 模型有自定义 layer（如 Vision-Language Projector）
- 模型发布时间早于 transformers 最新稳定版

**什么时候可以不设**：

- architectures 是标准类名（如 `Phi3ForCausalLM`、`LlamaForCausalLM`）
- transformers 明确公告支持（如 Qwen2.5 在 v4.50+）

> 🔧 某 35B MoE 模型的 modeling 代码引入 `qwen3_5_moe`，必须 transformers 5.x + trust\_remote\_code。用 transformers 4.x 时 auto\_map 找不到类，必须升级。

## 本地模型加载

模型文件在本地目录时的注意事项：

```python
# 标准做法
model = AutoModel.from_pretrained('/path/to/model', local_files_only=True)

# huggingface_hub 1.x 不认绝对路径
# 方案：monkey-patch
from huggingface_hub import validate_repo_id
validate_repo_id = lambda x, **kwargs: None
```

**路径特殊字符处理**：

| 字符      | 问题                                                     | 方案                             |
| :------ | :----------------------------------------------------- | :----------------------------- |
| 连字符 `-` | 目录转 module 名时 `Keye-VL-2` → `Keye_VL_2` vs `Keye_VL.2` | transformers 4.57.0+ 自动处理；手动改名 |
| 点号 `.`  | `locateanything.3b` → Python 解析为扩展名                    | `ln -s` 创建无点号软链接               |
| 空格      | hf-mirror 目录名含空格 → `from_pretrained` 失败                | 重命名目录                          |

> 🔧 Keye-VL-2 的目录名含连字符，transformers 4.40 在 import `transformers_modules.Keye-VL-2` 时失败。升级到 4.57.0+ 后解决。

### 终极方案：绕过 huggingface\_hub

当 `from_pretrained()` 完全不可用时（如自定义框架模型）：

```python
import safetensors.torch
import json

weights = safetensors.torch.load_file('model.safetensors')
with open('config.json') as f:
    config = json.load(f)

model = ModelClass(**config)
model.load_state_dict(weights)
```

## 量化加载与 quant\_config

不同量化方案的 `from_pretrained` 参数组合：

| 方案                | 核心参数                                                       | 依赖库          |
| :---------------- | :--------------------------------------------------------- | :----------- |
| bitsandbytes NF4  | `load_in_4bit=True, bnb_4bit_compute_dtype=torch.bfloat16` | bitsandbytes |
| bitsandbytes INT8 | `load_in_8bit=True`                                        | bitsandbytes |
| GPTQ（已量化）         | 直接加载 `.gptq` 目录                                            | auto-gptq    |
| AWQ（已量化）          | 直接加载 `.awq` 目录                                             | autoawq      |

```python
# bitsandbytes 4-bit + 双重量化 + NF4 + bfloat16 计算
model = AutoModel.from_pretrained(
    'model-path',
    load_in_4bit=True,
    bnb_4bit_quant_type='nf4',        # NF4 vs FP4
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,   # 双重量化，再省 ∼0.4GB/7B
    device_map='auto'
)
```

> 🔧 bge-reranker-v2-m3 用 float16 推理时 logit 溢出 → NaN，Cross-Encoder 必须用 float32。

## Tokenizer

### apply\_chat\_template

```python
messages = [{'role': 'user', 'content': [{'type': 'text', 'text': prompt}]}]

tokenized = tokenizer.apply_chat_template(
    messages,
    add_generation_prompt=True,
    return_tensors='pt'
)

# transformers 5.x 返回 BatchEncoding，不是 tensor
# 部分 tokenizer 直接返回 tensor，加判断（返回类型差异见 00 §七「tokenizer 返回对象的差异」）
if hasattr(tokenized, 'input_ids'):
    input_ids = tokenized['input_ids']
else:
    input_ids = tokenized
```

**chat\_template 不存在时的降级**：

```python
# 如果 tokenizer 没有 chat_template
if tokenizer.chat_template is None:
    # 用原始方式拼接（各模型格式不同）
    prompt = f"<|im_start|>user\n{prompt}<|im_end|>\n<|im_start|>assistant\n"
    input_ids = tokenizer.encode(prompt, return_tensors='pt')
```

## generate() 参数

### 参数推荐值（按任务）

| 任务    | temperature | top\_p | repetition\_penalty | max\_new\_tokens |
| :---- | :---------: | :----: | :-----------------: | :--------------: |
| 代码生成  |     0.2     |  0.95  |         1.0         |     512-1024     |
| 翻译/摘要 |     0.3     |  0.90  |         1.0         |      256-512     |
| 客服对话  |     0.6     |  0.90  |         1.05        |     512-1024     |
| 创意写作  |     0.9     |  0.92  |         1.15        |       1024+      |

### 常见陷阱

```python
# eos_token_id 可能是列表（多条停止条件，见 00 §七「eos_token_id 可能是列表」）
config.json: "eos_token_id": [2, 24]

# 正确传递
model.generate(..., eos_token_id=model.config.eos_token_id)

# pad_token_id 缺失 → 显存浪费
if tokenizer.pad_token is None:
    tokenizer.pad_token = tokenizer.eos_token  # 或自定义 [PAD]

# v5 后 max_length 弃用，必须用 max_new_tokens
# ❌ model.generate(..., max_length=256)
# ✅ model.generate(..., max_new_tokens=256)
```

> 🔧 eos\_token\_id 是列表时只传第一个值，导致模型在第一个停止条件后不停止，输出超长。必须传 `model.config.eos_token_id` 而不是 `model.config.eos_token_id[0]`。

## 数据类型

各精度关键参数对比：

|    精度    |   每参数大小  | 显存节省 vs FP32 | 数值范围         | 推荐场景              |
| :------: | :------: | :----------: | :----------- | :---------------- |
|  float32 |  4 bytes |      基准      | ±3.4×10³⁸    | Cross-Encoder 重排序 |
| bfloat16 |  2 bytes |      50%     | ±3.4×10³⁸    | 通用推理              |
|  float16 |  2 bytes |      50%     | ±65504       | 不推荐（溢出风险）         |
|   int8   |  1 byte |      75%     | \[-128, 127] | 推理加速              |
|   int4   | 0.5 byte |     87.5%    | \[-8, 7]     | 显存极度受限            |

```python
# processor 返回 float32 pixel_values，模型可能是 bfloat16
pixel_values = pixel_values.to(model.dtype)

# Cross-Encoder 重排序用 float32，不要用 float16
# float16 最大 65504，高相关度 logit 溢出 → NaN
model = AutoModel.from_pretrained('reranker', torch_dtype=torch.float32)

# generate() 返回值可能是专用对象而非 tensor
# 扩散模型返回 DiffusionGemmaGenerationOutput
output = model.generate(...)
# 确认输出类型：dir(output)
print(type(output))  # 可能是特殊类
# 用 .sequences / .text / 等属性访问
```

## LoRA Adapter 合并与部署

微调后的 LoRA adapter 部署到生产环境有两种方式：

| 方式            | 做法                                    | 优点                      | 缺点           |
| :------------ | :------------------------------------ | :---------------------- | :----------- |
| **online 加载** | base + PeftModel.from\_pretrained     | 不修改 base 权重，随时换 adapter | 加载慢，多一层包装    |
| **merge 后部署** | merge\_and\_unload → save\_pretrained | 推理速度快，无额外依赖             | base 被修改，不可逆 |

```python
# merge 方式（推荐生产部署）
from peft import PeftModel

base_model = AutoModel.from_pretrained('base-model-path')
peft_model = PeftModel.from_pretrained(base_model, './lora-adapter')
merged_model = peft_model.merge_and_unload()

# 保存合并后的模型（结构同原始模型，权重已合并）
merged_model.save_pretrained('./merged-model')
tokenizer.save_pretrained('./merged-model')

# 之后用标准方式加载
model = AutoModel.from_pretrained('./merged-model')
```

**合并后的目录结构应包含**：

```
merged-model/
├── config.json              # 必须
├── model.safetensors        # 合并后的权重
├── tokenizer_config.json    # 必须
├── tokenizer.json
├── special_tokens_map.json
└── generation_config.json
```

> 注意：merge 后无法"还原"回未微调状态。生产部署建议：先保存一份 base 模型备份，再 merge。

## 多模态模型加载

多模态模型（视觉-语言、音频等）需要 processor + model 双加载：

```python
from transformers import AutoProcessor, AutoModelForCausalLM

processor = AutoProcessor.from_pretrained('model-path', trust_remote_code=True)
model = AutoModelForCausalLM.from_pretrained(
    'model-path',
    device_map='auto',
    torch_dtype=torch.bfloat16,
    trust_remote_code=True
)

# 多模态推理（VL 模型示例）
from PIL import Image
image = Image.open('test.png')
messages = [
    {'role': 'user', 'content': [
        {'type': 'image', 'image': image},
        {'type': 'text', 'text': '描述这张图片'}
    ]}
]
inputs = processor.apply_chat_template(
    messages,
    add_generation_prompt=True,
    return_tensors='pt'
).to(model.device)
```

**常见问题**：

| 问题                                | 原因                               | 方案                                                           |
| :-------------------------------- | :------------------------------- | :----------------------------------------------------------- |
| processor 无 `apply_chat_template` | processor 类不同                    | 手动拼接 prompt + `processor(images, text, return_tensors='pt')` |
| pixel\_values dtype 不匹配           | processor 返回 float32，模型 bfloat16 | `pixel_values = pixel_values.to(model.dtype)`                |
| multi-modal projector 缺失          | 模型文件不完整                          | 检查目录是否有 `projector.bin` 等额外权重                                |

> 🔧 非标准多模态架构（如 Keye-VL-2、某扩散 MoE）的 modeling 代码常硬 import `flash_attn`。如果部署平台不支持编译 flash\_attn，必须选预装 flash\_attn 的基础镜像。

## 加载后验证

无论哪种加载方式，加载后都应做最小验证：

```python
def validate_model(model, tokenizer):
    """加载后快速验证模型是否正常工作"""
    # 1. dtype 检查
    param_dtype = next(model.parameters()).dtype
    print(f"Model dtype: {param_dtype}")

    # 2. device 检查
    param_device = next(model.parameters()).device
    print(f"Model device: {param_device}")

    # 3. 输出 sanity check
    test_input = tokenizer("Hello", return_tensors='pt').to(model.device)
    with torch.no_grad():
        output = model.generate(**test_input, max_new_tokens=10)
    print(f"Output: {tokenizer.decode(output[0])}")

    # 4. 显存检查
    if torch.cuda.is_available():
        print(f"VRAM: {torch.cuda.memory_allocated() / 1024**3:.2f} GB")
```

## 常见 Import 错误

| 错误                                          | 原因                              | 方案                                                 |
| :------------------------------------------ | :------------------------------ | :------------------------------------------------- |
| `all_special_tokens_extended`               | transformers 5.x 移除了            | `pip install transformers==4.48.0` 或加 monkey-patch |
| `LossKwargs` 找不到                            | modeling 文件与 transformers 版本不匹配 | 不设 trust\_remote\_code，升级 transformers             |
| `flash_attn` 编译失败                           | 部署平台 32GB RAM 不足 + 相关源码站点被墙 | 用预装 flash\_attn 的镜像                                |
| 模型目录名含 `.` 不能 import                        | Python 解析 `.2` 为扩展名             | `ln -s` 创建无点号软链接                                   |
| 目录名含连字符不能 import                            | Python 解析为减号                    | 升级 transformers 4.57.0+                            |
| `KeyError: 'llama'`                         | transformers 版本太老不识别该模型         | 升级 transformers                                    |
| `CUDA out of memory` 未到 100%                | device\_map 分配不均                | 加 max\_memory + CPU offload                        |
| `RuntimeError: CUDA error: no kernel image` | CUDA 版本与编译的算子不匹配                | 确认 CUDA 兼容性表                                       |
