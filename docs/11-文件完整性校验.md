# 11 · 文件完整性校验

本文是专项之一：覆盖下载与加载环节的完整性校验，以及供应链安全防护。大模型权重文件体积大、下载链路长，损坏或被篡改往往在加载时才暴露，因此校验应作为独立步骤前置处理。

## 为什么需要校验

大模型权重文件通常体积巨大（几个 GB 到几十个 GB），下载过程中可能：

- 网络中断导致文件不完整但进程正常退出（exit 0）
- 分片文件（shard）部分损坏
- 磁盘空间不足导致写入不完整
- 镜像站点中转错误导致文件字节翻转

损坏的模型文件加载时表现各异：有时报 `key not found`，有时模型加载成功但输出异常。

> 🔧 **门控模型特殊陷阱**：门控模型（需授权的模型）通过镜像站点不带 token 下载时，文件大小正常但权重是随机占位数据，模型加载不报错但输出乱码。选模型前先确认模型页是否有授权按钮。

### 供应链安全背景

HuggingFace Hub 上已有多次恶意模型仓库事件（如 2026 年 5 月的 `Open-OSS` 钓鱼仓库事件，24 万次下载）。攻击者通过以下方式注入恶意代码：

| 攻击向量 | 描述 | 防御方法 |
|:--------|:----|:--------|
| **Typosquatting（域名抢注）** | 注册与合法模型名差一个字符的组织/仓库名 | 使用前校验 `model_info()` 中的正式哈希值 |
| **Pickle 反序列化 RCE** | `.pt`/`.pth`/`.bin` 文件是 pickle 格式，可嵌入任意 Python 代码 | 优先用 `.safetensors`，禁用 `trust_remote_code=True` |
| **Trojanised（木马化）** | 同仓库中同时放合法 safetensors 和恶意 .bin 文件 | 使用前确认只加载预期格式的权重文件 |
| **恶意辅助脚本** | README 指导用户运行 `scripts/setup.py`，实际下载恶意二进制 | 审查 README 中的每条命令和脚本内容 |

> **结论**：完整性校验不仅是检查文件是否损坏，也是防御供应链攻击的第一道防线。

## 校验方法

### 1. 官方 SHA256 校验（最可靠）

从 HuggingFace 模型页获取官方哈希值，下载后本地比对：

| 获取方式 | 适合场景 |
|:--------|:--------|
| 模型页 `Files` → 点 `⋯` → `Copy file hash` → `SHA256` | 手动下载 |
| `README.md` 中 `Verification` 小节 | 手动下载 |
| `huggingface_hub.model_info()` API | 自动化流水线 |

```python
# 自动化哈希校验
from huggingface_hub import hf_hub_download, model_info
import hashlib

model_id = "meta-llama/Llama-2-7b-hf"
filename = "model-00001-of-00002.safetensors"

# 1. 从 HF API 获取官方哈希
info = model_info(model_id)
expected_hash = None
for sibling in info.siblings:
    if sibling.rfilename == filename:
        expected_hash = sibling.lfs.sha256  # HF 后端记录的 SHA256
        break

# 2. 下载文件
local_path = hf_hub_download(
    repo_id=model_id,
    filename=filename,
    revision="main",
)

# 3. 计算本地哈希并比对
with open(local_path, 'rb') as f:
    actual_hash = hashlib.sha256(f.read()).hexdigest()

assert actual_hash == expected_hash, \
    f"SHA256 mismatch: {actual_hash[:16]}... != {expected_hash[:16]}..."
print("✔ 哈希校验通过")
```

> **注意**：`model_info()` API 对未认证请求有频率限制（15 req/min），批量验证多个模型时需加缓存。

### 2. 分片文件优先检查策略

```python
from pathlib import Path

model_dir = Path("/path/to/model")

# 按文件大小排序：分片文件最大，优先检查
shard_patterns = ["*.safetensors", "pytorch_model*.bin"]
for pattern in shard_patterns:
    for shard in sorted(model_dir.rglob(pattern), key=lambda f: f.stat().st_size, reverse=True):
        size_gb = shard.stat().st_size / 1024**3
        if size_gb > 1:  # > 1GB 的分片文件优先检查
            print(f"检查分片: {shard.name} ({size_gb:.1f}GB)")
            try:
                # 对 safetensors 做结构测试
                weights = safetensors.safe_open(str(shard), framework="pt")
                keys = weights.keys()
            except Exception as e:
                print(f"⚠ 分片损坏: {shard.name} → {e}")
```

### 3. safetensors 打开测试

```python
import safetensors

try:
    weights = safetensors.safe_open('model.safetensors', framework='pt')
    keys = weights.keys()
    print(f'文件正常，包含 {len(keys)} 个 key')
except RuntimeError as e:
    # safetensors >= 0.4.0 会输出损坏位置和期望字节数
    print(f'文件损坏：{e}')
    # 建议兜底加载 .bin 格式
    print('尝试加载 .bin 格式...')
    model = AutoModel.from_pretrained('./model', use_safetensors=False)
```

> 🔧 **实战经验**：分片文件（model-00002-of-00003.safetensors 等）最容易损坏，加载时报 `pytorch_model.bin.index.json key not found`。怀疑文件损坏时先检查对应分片文件大小是否接近预期值。

### 4. 加载验证

```python
try:
    model = AutoModel.from_pretrained('/path/to/model', local_files_only=True)
    # 做一次简单推理确认输出正常
    test_output = model.generate(input_ids, max_new_tokens=10)
except Exception as e:
    print(f'模型加载验证失败：{e}')
```

### 5. snapshot_download 自动校验

`huggingface_hub` 的 `snapshot_download()` 内置 SHA256 自动校验：

```python
from huggingface_hub import snapshot_download

# 自动下载并校验所有文件的 SHA256
model_path = snapshot_download(
    repo_id="meta-llama/Llama-2-7b-hf",
    cache_dir="./models",
    resume_download=False,     # 强制重新下载已损坏的文件
    force_download=False,      # False = 仅重新下载校验失败的文件
)
```

> **注意**：如果 `resume_download=True`，`snapshot_download` 会跳过已存在的文件而不校验。对于生产环境建议设为 `False`。

## 下载过程中的常见问题

| 现象 | 原因 | 方案 |
|:----|:----|:-----|
| 镜像站点下载 1h+ 后 `403 Forbidden` | XetHub CAS bridge 预签名 URL 仅 1 小时有效，网速慢时超时 | 分片下载或多次重试 |
| 🔧 **modelscope 下载失败留下 `.incomplete` 文件** | 哈希命名的临时文件，无法恢复也与其他工具不互通 | 清缓存用镜像站点重下 |
| 🔧 **`snapshot_download` exit 0 但文件不完整** | XetHub CAS bridge 静默退出，无异常无日志 | 用 `list_repo_files` + 逐个 `hf_hub_download` + 重试机制替代 |
| 🔧 **门控模型用镜像站点下载不报错但输出乱码** | 不带 token 下载的权重是随机占位数据（文件大小正常但内容垃圾） | 带 HF Token 重新下载 |
| 🔧 **Windows 下重下失败** | `.incomplete` 或 `.tmp` 缓存文件被占用阻塞 | `Get-Process python \| Stop-Process -Force` 杀进程 + 手动删除 `%USERPROFILE%\.cache\huggingface\hub` 再试 |
| 🔧 **网络不稳时 snapshot_download 全白费** | 并行多文件，一个 403 全部中断重来 | 用 `list_repo_files` → 逐个 `hf_hub_download` + `resume_download=True` + 指数退避重试 |
| 🔧 **Windows 上不要设 `HF_HUB_ENABLE_HF_TRANSFER=1`** | 依赖 Rust 编译的 `hf_transfer` 二进制，Windows 上几乎肯定没有，设了反而引入 SSL 异常 | 不用此环境变量 |
| 模型页显示 5GB 但下载后校验不通过 | 镜像站点中转时字节翻转或截断 | 用官方 SHA256 校验，不从第三方镜像加载 |
| safetensors 加载报错但 .bin 可用 | safetensors 文件损坏 | 回退 `use_safetensors=False`，或删除缓存强制重下 |

## 最佳实践

1. **下载完成后立即校验**：不要等加载时才发现文件损坏
2. **损坏文件立即删除**：残留的 `.incomplete` / `.lock` 文件会阻塞重试
3. **重跑前清缓存**：包括 `.cache/huggingface/download/` 下的 `.incomplete` 文件
4. **保留小文件**：tokenizer/config 等小文件不易损坏，重下载时跳过已存在的完整小文件
5. **curl -C - 断点续传**：大文件用 curl 配合 `-C -` 做断点续传，多次中断也能最终完成
6. **分片文件优先检查**：`snapshot_download` 对 >5GB 的分片文件最容易下载不完整，怀疑损坏时先检查分片文件大小
7. **生产环境用 `snapshot_download` 而非 `from_pretrained`**：前者内置校验，后者依赖缓存机制
8. **为门控模型显式设置 `HF_TOKEN`**：使用前检查环境变量是否存在，不存在则报错，避免静默下载垃圾权重
9. **🔧 下载脚本用 try/except 兜底**：主函数外层必加 `try: main() except: log(traceback.format_exc())`，否则静默崩溃时日志什么都看不到
10. **🔧 网络不稳时不用 snapshot_download**：用 `list_repo_files` → 逐个 `hf_hub_download` + 指数退避（每次等待 ×2，上限 10 分钟）+ 30 次重试
11. **🔧 通宵下载脚本模板**：先清缓存残留 → 先下小文件（config/tokenizer）→ 再下大权重 → 全程写日志 → 已存在文件跳过
