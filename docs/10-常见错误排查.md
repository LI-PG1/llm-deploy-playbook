# 10 · 常见错误排查

本文是主线的兜底速查手册：将部署各环节可能遇到的问题按类别汇总，便于定位时直接检索。正文章节（01–09）覆盖标准流程，本篇覆盖异常与边界情况。

## 构建/安装失败

| 问题                        | 原因                                               | 方案                                                                 |
| :------------------------ | :----------------------------------------------- | :----------------------------------------------------------------- |
| 层数超出上限                    | RUN 太多                                           | 用 `&&` 合并                                                          |
| CUDA 版本不匹配                | 基础镜像 CUDA ≠ torch 需要的 CUDA                       | 对齐版本                                                               |
| tar.gz 编译永久卡死（CPU=0%）     | setup.py 触发外网下载，网络不通                             | 用 `--no-deps` 或换 whl                                               |
| apt-get 卡死                | apt 源不可达                                         | 换阿里云源                                                              |
| pip 静默失败                  | 内网源没有这个包                                         | 指定公网源                                                              |
| `ModuleNotFoundError` 编译后 | `--no-deps` 缺少子依赖                                | 单独补装缺失的包                                                           |
| Dockerfile heredoc 解析错误   | `cat > file << 'EOF'` 中 Python 关键词被 Docker 解析器误解 | 用 `printf` 替代：`printf 'import torch\ndef f(x): return x\n' > file` |
| flash\_attn 编译 OOM        | 部署平台 32GB RAM + 相关源码站点被墙，编译不可行                 | 选预装 flash\_attn 的基础镜像                                              |
| 不可编译包缺失                   | 包源码在 GitHub 404，pip 编译 tar.gz 永久卡死               | 创建假包目录 + stub，用 `PYTHONPATH` 指向假包                                  |
| pip config set 全局源导致失败    | 一个源不存在某包时所有 `pip install` 都失败                    | 每个 `RUN pip install` 显式指定 `--index-url`                            |
| 外网 PyPI 源不可达              | `pypi.nvidia.com` / `download.pytorch.org` 等被墙   | 用内网源或公网源                                                           |

### Docker 构建缓存层命中原则

理解 Docker 缓存机制能减少 50% 的构建时间：

```
# ❌ 不好的顺序：代码变了，前面的依赖安装层缓存全部失效
COPY . .
RUN pip install -r requirements.txt

# ✅ 好的顺序：只当 requirements.txt 变化时才重装依赖
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
```

| 原则          | 说明                                                                             |
| :---------- | :----------------------------------------------------------------------------- |
| 稳定指令前置      | `FROM` → `WORKDIR` → `COPY requirements.txt` → `RUN pip install` → `COPY code` |
| 变化频率越低越靠前   | 基础镜像 > 系统依赖 > Python 依赖 > 代码                                                   |
| 文件内容变化中断缓存链 | `COPY` 的文件变了，该层及之后所有层缓存失效                                                      |
| 缓存失效后尽量合并   | 失效后的多条 `RUN` 合并为一条，减少层数                                                        |

### 镜像瘦身常见手法

| 手法                       | 效果                                                | 适用场景                                    |
| :----------------------- | :------------------------------------------------ | :-------------------------------------- |
| 多阶段构建                    | 分离编译环境和运行环境，镜像可缩小 60-80%                          | 有编译步骤的 Python/C++ 项目                    |
| 删除包管理器缓存                 | `rm -rf /var/lib/apt/lists/*` + `pip cache purge` | 所有 Dockerfile                           |
| 用 `--no-cache-dir` 装 pip | 省去 wheel 缓存                                       | pip install                             |
| 不要 `apt-get upgrade`     | 会拉入大量新包                                           | 非安全更新不需要                                |
| `.dockerignore` 排除无关文件   | 防止 `COPY .` 带入大文件                                 | `**/*.pyc`, `**/.git`, `**/__pycache__` |

> **核心原理**：Docker 分层是"只增不减"的。`RUN rm -rf /var/lib/apt/lists/*` 只是隐藏了文件，数据仍在上层。安装+清理必须在**同一条 RUN** 中完成才能真正减小体积。

## 加载/运行错误

> 下表涉及 tokenizer 返回差异、`eos_token_id` 为列表、FP8 反量化等概念，原理统一见 [00-前置知识](00-前置知识.md) §七。

| 问题                                                                     | 原因                                                                                                | 方案                                                                                              |
| :--------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------ | :---------------------------------------------------------------------------------------------- |
| OOM                                                                    | 显存不足                                                                                              | 缩短上下文、降显存利用率、加多卡                                                                                |
| `HFValidationError`                                                    | huggingface\_hub 1.x 不认本地路径                                                                       | monkey-patch 或 safetensors 直接加载                                                                 |
| 模型加载成功但输出乱码                                                            | 门控模型无 token 下载了垃圾数据                                                                               | 带 HF Token 重新下载                                                                                 |
| `key not found` 加载权重                                                   | safetensors 文件损坏                                                                                  | 删除损坏文件重新下载                                                                                      |
| apply\_rotary\_emb shape 不匹配                                           | flash\_attn 版本与模型 modeling 不兼容                                                                    | 自定义 reshape 替换                                                                                  |
| `device_map` ImportError                                               | 镜像缺 accelerate                                                                                    | 小模型用 `model.cuda()` 替代                                                                          |
| check\_model\_inputs 报错                                                | transformers 版本变了内部 API                                                                           | 加兼容 patch                                                                                       |
| safetensors 版本冲突                                                       | 镜像 safetensors 版本太低                                                                               | pip 升级                                                                                          |
| validate\_repo\_id 拒绝绝对路径                                              | huggingface\_hub 1.x 只认 `namespace/repo-name` 格式                                                  | app.py 首行 monkey-patch `validate_repo_id = lambda x, **kwargs: None` + `TRANSFORMERS_OFFLINE=1` |
| 模型名含连字符 import 失败                                                      | 目录名 `Keye-VL-2` 中连字符被解析为减号                                                                        | 升级 transformers 到 4.57.0+                                                                       |
| check\_model\_inputs 改为装饰器                                             | transformers 4.57 从普通函数改为装饰器                                                                      | 定义 `_patched(func=None)` 兼容两种调用方式                                                               |
| Gradio Chatbot 的 type 参数移除                                             | Gradio 5.x/6.x 使用 dict 格式而非 tuples                                                                | return `[{"role": "user", "content": "..."}, {"role": "assistant", "content": "..."}]`          |
| tokenizer 返回 tensor 而非 BatchEncoding                                   | MiniCPM 等 `return_tensors="pt"` 直接返回 Tensor                                                       | 加 `hasattr(tokenized, "input_ids")` 判断分支                                                        |
| diffusers 旧 PyTorch 无 torch.xpu                                        | diffusers 0.38 加载用到 torch.xpu，PyTorch 2.2.2 没有                                                    | 选 CUDA12.4-PyTorch-2.6.0 基础镜像                                                                   |
| pip install --no-deps 导致子依赖缺失                                          | gradio 6.x 依赖 click/semver/tomli 等子包不安装                                                           | 去掉 `--no-deps`，让 pip 自动解析完整依赖链                                                                  |
| 多 GPU 需 torchrun 启动                                                    | 部署平台多 GPU RPC 模式下 python app.py 触发冲突                                                                | `torchrun --nproc_per_node=1 --nnodes=1` 启动                                                     |
| 🔧 **`trust_remote_code=True`** **与内置实现冲突**                            | 模型目录的 `modeling_xxx.py` 引用了新版 transformers 才有（如 `LossKwargs`），旧版找不到                               | 看 `config.json`：`architectures` 是标准类名（phi3/llama/qwen2）就**不设** trust\_remote\_code，用内置实现        |
| 🔧 **transformers 5.x** **`apply_chat_template`** **返回 BatchEncoding** | 5.x 返回 `BatchEncoding` 对象而非 tensor，`tokenized.shape` 报 AttributeError                             | 用 `tokenized["input_ids"]` 取长度，`model.generate(**tokenized)` 拆包传入                               |
| 🔧 **模型内部 Conditioner 类从 HuggingFace 下载权重**                            | `T5GemmaConditioner.__init__` 内部调用 `from_pretrained()` 从远程下载                                      | monkey-patch 整个 `__init__`，用 `local_files_only=True` + 本地路径                                     |
| 🔧 **非标准框架的** **`from_pretrained()`** **需在多个模块同时 patch**               | 某些模型（如 stable\_audio\_3）在 `model_configs`、`huggingface_hub`、`transformers` 多处调用 `hf_hub_download` | 遍历所有可能的调用点逐个替换 patch                                                                            |
| 🔧 **`eos_token_id`** **为列表时需完整传递**                                    | 部分模型 config 中 `eos_token_id: [2, 24]`，只传标量导致模型过早终止                                                | `generate()` 中 `eos_token_id=model.config.eos_token_id`                                         |
| 🔧 **模型目录名含点号（`.`）导致 Python import 失败**                                | `model-v2.2` 中 `.2` 被误解析为文件扩展名，trust\_remote\_code 时报 `ModuleNotFoundError`                       | start.sh 中 `ln -sf` 建无点号软链接 + `export MODEL_PATH` 指向新路径                                         |
| 🔧 **≤3B 模型不需要** **`device_map="auto"`**                               | 触发 accelerate 依赖（镜像里可能没装或不兼容）                                                                     | 直接用 `model.cuda()`，行数更少、更快、零依赖                                                                  |
| 🔧 **部署平台空闲超时（3600s）会杀死长耗时进程**                                           | 无 HTTP 活动 1h 后进程被 kill                                                                            | start.sh 先启动 Gradio 再后台加载模型 + 保活线程每 5min 发 HTTP 请求到 localhost:7860                              |

### OOM 排查步骤

```
OOM 发生时：
  Step 1 ── 看当前显存占用：nvidia-smi
  Step 2 ── 看有无残留进程：ps aux | grep python（杀掉）
  Step 3 ── 确认 vLLM gpu_memory_utilization（默认 0.9）
  Step 4 ── 确认 ECC 是否启用（ECC 可吃掉 12% 显存）
  Step 5 ── 确认系统预留显存：nvidia-smi -i 0 -q -d MEMORY
  Step 6 ── 如果还有剩余但 vLLM OOM：降低 gpu_memory_utilization
  Step 7 ── 还是不够：启用 TP/PP 多卡并行
```

> **ECC 显存陷阱**：H100/A100 启用 ECC 后会预留约 12% 显存，vLLM 探测不到这部分。如果默认 gpu\_memory\_utilization=0.9，系统预留 12% + model weights，可能实际只有 78% 可用。公式：`有效利用率 = gpu_memory_utilization × (1 - 系统预留比例)`。

### 响应质量问题排查（同步 vLLM v0.8.0 行为变更）

从 vLLM v0.8.0 起，默认采样参数来源从 vLLM 的默认值**改为模型自带的** **`generation_config.json`**。

| 现象              | 排查方法                                                   |
| :-------------- | :----------------------------------------------------- |
| 升级 vLLM 后输出质量下降 | 用 `--generation-config vllm` 切回旧默认值对比                  |
| 输出风格突变          | 检查 `generation_config.json` 中的 temperature/top\_p 是否合理 |
| 某个模型输出异常好/差     | 对比 vLLM 默认值与模型默认值的差异                                   |

| 冲突                                           | 方案                                                         | <br />                                                                     |
| :------------------------------------------- | :--------------------------------------------------------- | :------------------------------------------------------------------------- |

## 推理异常

| 问题                                                                | 方案                                                                                                             | <br />                                                                                                                                                |
| :---------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------- |
| 输出质量差                                                             | 检查量化精度、确认 prompt 模板匹配模型                                                                                        | <br />                                                                                                                                                |
| 推理慢                                                               | 开 vLLM、缩短上下文、确认模型在 GPU 上                                                                                       | <br />                                                                                                                                                |
| 重复输出                                                              | 调高 temperature、调 `repetition_penalty`                                                                          | <br />                                                                                                                                                |
| 数学/事实类问题出错                                                        | 用 greedy decoding（temperature=0），放弃 repetition\_penalty                                                        | <br />                                                                                                                                                |
| 🔧 **repetition\_penalty 对数学推理有副作用**                              | 小模型数学推理时出现 token 级随机错误                                                                                         | 数学/事实类问题用 temperature=0（greedy decoding）最可靠                                                                                                           |
| 🔧 **重排序输出都是 NaN**                                                | bge-reranker 等 Cross-Encoder 用 float16，logit 范围极大溢出为 inf → NaN                                                 | 改 float32，560M 模型仅需 \~4.4GB 显存                                                                                                                        |
| 🔧 **扩散模型 build\_schedule 类型错误**                                  | `n_points`/`steps` 参数传了 float，`torch.linspace` 要求 int                                                          | 在 monkey-patch 函数入口做 `int(n_points)`、`int(steps)` 转换                                                                                                  |
| 🔧 **vLLM + transformers 5.x + Qwen2 架构需 monkey-patch tokenizer** | vLLM `get_cached_tokenizer()` 调用 `tokenizer.all_special_tokens_extended`，transfomers 5.x Qwen2Tokenizer 已移除该属性 | import vLLM 前：`if not hasattr(Qwen2Tokenizer, 'all_special_tokens_extended'): Qwen2Tokenizer.all_special_tokens_extended = property(lambda self: [])` |

## 网络与多机部署

| 问题                            | 原因                     | 方案                                                                  |
| :---------------------------- | :--------------------- | :------------------------------------------------------------------ |
| 模型在共享文件系统加载慢                  | NFS/distributed FS 延迟高 | 模型放到本地磁盘                                                            |
| NCCL `unhandled system error` | 多节点网络不通                | 检查安全组、VPC、防火墙规则                                                     |
| vLLM 获取 IP 错误                 | 多网卡环境 IP 不匹配           | `export VLLM_HOST_IP=<your_ip>`                                     |
| NCCL/GLOO 网口错误                | 默认网口不是数据通信口            | `export NCCL_SOCKET_IFNAME=eth0` + `export GLOO_SOCKET_IFNAME=eth0` |
| 节点间 curl 成功但 MPI 通信失败         | 防火墙规则限制非 HTTP 端口       | 打开节点间全部 TCP 端口                                                      |

## vLLM 调试工具

当 vLLM 卡死或崩溃时，用以下环境变量逐步定位：

```bash
# 1. 先开 DEBUG 日志
export VLLM_LOGGING_LEVEL=DEBUG

# 2. 如果卡在 CUDA，用 CUDA_LAUNCH_BLOCKING 找哪个 kernel 出问题
export CUDA_LAUNCH_BLOCKING=1

# 3. 如果卡在 NCCL 通信
export NCCL_DEBUG=TRACE

# 4. 极端情况：记录所有函数调用（会显著减慢启动速度）
export VLLM_TRACE_FUNCTION=1

# 5. 如果 CUDAGraph 报错（self.graph.replay() 附近）
# 添加 --enforce-eager 关闭 CUDAGraph，找到具体 CUDA 操作
```

## Python 多进程注意事项

vLLM 内部使用多进程，以下代码会触发死锁：

```python
# ❌ 错误：__main__ 之外创建 LLM 实例
import vllm
llm = vllm.LLM(...)  # 多进程 fork 时重复创建

# ✅ 正确：在 if __name__ 块内创建
from vllm import LLM

if __name__ == '__main__':
    llm = LLM(model="meta-llama/Llama-2-7b")
    output = llm.generate("Hello!")
```

**规则**：任何使用 vLLM、Ray、torchrun 等并行框架的代码，LLM 实例创建必须在 `if __name__ == '__main__'` 块内。

## 常见问题一览

| 现象                                                      | 原因                                       | 解决                                                              |
| :------------------------------------------------------ | :--------------------------------------- | :-------------------------------------------------------------- |
| `ModuleNotFoundError: No module named 'app'`            | benchmark.py 和 app.py 不在同一目录             | 把 benchmark.py 放到 code/ 目录                                      |
| `CUDA out of memory`                                    | app.py 未完全退出                             | `ps aux \| grep python` 杀掉残留进程                                  |
| 第一次 TTFT 比后面高 2-3 倍                                     | 冷启动（GPU kernel 编译）                       | 预热机制已内置处理                                                       |
| 容器启动后反复重启                                               | 入口脚本 `app.py` 有未捕获异常                     | 在 `start.sh` 中用 `exec python app.py 2>&1` 捕获日志                  |
| tok/s 看不懂                                               | —                                        | 1 token ≈ 1.5 汉字，100 tok/s ≈ 150 字/秒                            |
| 不确定 MoE 模型显存需求                                          | —                                        | 看激活参数不是总参数。如 `122B-A10B` 按 10B 判断                               |
| 多节点部署 Ingress 卡住                                  | Ingress 对所有节点发起健康检查                      | 排查节点间网络连通性                                                      |
| 🔧 **Gradio 日志刷「Invalid HTTP request received」WARNING** | 部署平台反向代理发送 HTTP/1.0 无 Host 头给 uvicorn 健康检查 | `logging.getLogger("uvicorn.error").disabled = True` 压制警告，不影响功能 |
