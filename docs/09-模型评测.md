# 09 · 模型评测

部署完成后，以可复现的指标对模型做性能与质量评测，支撑量化对比、引擎对比与硬件选型决策。

## 指标体系

### 核心指标（6 维）

| 指标            | 中文           |    单位    | 测量方法                                | 意义           |
| :------------ | :----------- | :------: | :---------------------------------- | :----------- |
| **TTFT**      | 首 token 延迟   |    ms    | 输入完成 → 第一个 token 输出                 | 响应速度，交互式场景关键 |
| **TPOT**      | 每 token 生成时间 | ms/token | (总耗时 - TTFT) / (输出 token 数 - 1)     | 持续生成效率       |
| **ITL**       | token 间延迟    | ms/token | 相邻 token 输出时间间隔的均值                  | 流畅度感知        |
| **TPS**       | 每秒 token 数   |   tok/s  | 输出 token 总数 / 总耗时                   | 吞吐量，批量场景关键   |
| **RPS**       | 每秒请求数        |   req/s  | 并发请求总数 / 总耗时                        | 服务容量         |
| **VRAM Peak** | 峰值显存         |    GB    | `torch.cuda.max_memory_allocated()` | 最低 GPU 规格要求  |

> 🔧 **实战经验**：实际报告中聚焦 4 个核心指标（TTFT / E2E / 吞吐 / P50-P95），TPOT 和吞吐高度相关，RPS 在串行测试中无意义。减少指标数量让决策更清晰。

### 附加指标

| 指标          | 说明           | 测量方式                    |
| :---------- | :----------- | :---------------------- |
| GPU 利用率     | 计算核心使用率      | `nvidia-smi` / `pynvml` |
| 预填充时间       | prefill 阶段耗时 | 服务端日志                   |
| KV Cache 显存 | 与模型权重分离      | vLLM 内部统计               |
| 冷启动时间       | 模型从加载到首次推理   | 时间戳差                    |

## 测量方法

### 预热与稳态

正式测量前必须预热，否则数据不可靠：

```python
import time
import torch

def warmup(model, tokenizer, n_warmup=10):
    """预热模型，消除冷启动影响"""
    sample = "Hello, world!"
    for _ in range(n_warmup):
        inputs = tokenizer(sample, return_tensors='pt').to(model.device)
        _ = model.generate(**inputs, max_new_tokens=32)
        torch.cuda.synchronize()  # 等待 GPU 完成

# 预热后休息一下（cooldown）
time.sleep(2)
```

|  预热次数 | 适用场景          |
| :---: | :------------ |
|  3-5  | 小模型（<7B），快速对比 |
| 10-20 | 标准评测          |
|  50+  | 生产环境精度测试      |

> 🔧 **实战经验**：benchmark 脚本用 1 次预热（生成 10 个 token），预热 TTFT 单独记录在 JSON 的 `预热数据` 字段，明确标为"仅供参考"。跑某些模型时首次 TTFT 是稳态的 3 倍，预热数据计入统计会导致报告完全不可信。

### 统计口径

不要只报均值，业界标准是报分布：

| 口径           | 含义       | 推荐场景   |
| :----------- | :------- | :----- |
| **P50（中位数）** | 一半请求快于它  | 日常体验感知 |
| **P95**      | 5% 请求慢于它 | 长尾优化   |
| **P99**      | 1% 请求慢于它 | SLA 保障 |
| 均值           | 总体平均     | 仅粗估    |

### 输入长度分档

不同输入长度对性能影响很大，应分档测量：

| 分档     |     输入长度    | 场景模拟  | 典型模型行为          |
| :----- | :---------: | :---- | :-------------- |
| Short  |  128 tokens | 简短问答  | TTFT 低，TPS 高    |
| Medium |  512 tokens | 一般对话  | 中等 TPS          |
| Long   | 2048 tokens | 文档总结  | KV Cache 显存开始显著 |
| XLong  | 8192 tokens | 长文档分析 | TTFT 高，显存压力大    |

> 🔧 **实战经验**：用中文 prompt 分 4 档 —— 短文本(\~50字) / 中文本(\~500字) / 长文本(\~2000字) / 超长文本(\~8000字，仅长上下文模型)。超长文本通过重复基准段落自动生成。长上下文自动检测逻辑按模型名关键词匹配。

### 并发 Profile

单次测量不能反映真实场景，至少测 4 个并发级别：

| 并发数 | 场景    | 关注指标         |
| :-: | :---- | :----------- |
|  1  | 单人体验  | TTFT, TPOT   |
|  4  | 轻量并发  | TPS, VRAM    |
|  16 | 中等负载  | RPS, GPU 利用率 |
|  64 | 高负载压测 | P95 延迟，稳定性   |

```python
import asyncio
import aiohttp
import time

async def benchmark_concurrent(url, prompts, concurrency=16):
    """并发压力测试"""
    semaphore = asyncio.Semaphore(concurrency)

    async def single_request(prompt):
        async with semaphore:
            async with aiohttp.ClientSession() as session:
                start = time.time()
                async with session.post(url, json={
                    'prompt': prompt,
                    'max_new_tokens': 256
                }) as resp:
                    result = await resp.json()
                elapsed = time.time() - start
                return elapsed

    tasks = [single_request(p) for p in prompts]
    latencies = await asyncio.gather(*tasks)
    return {
        'p50': sorted(latencies)[len(latencies)//2],
        'p95': sorted(latencies)[int(len(latencies)*0.95)],
        'avg': sum(latencies) / len(latencies)
    }
```

## 执行步骤

### 标准评测流程（7 步）

```
Step 1: 确定评测目标（单卡速度？多卡并发？量化对比？）
    ↓
Step 2: 固定控制变量（同模型、同精度、同 prompt 集、同批次）
    ↓
Step 3: 准备测试 Prompt 集（固定长度分档）
    ↓
Step 4: 预热模型（10 次）
    ↓
Step 5: 记录测量（至少 3 轮取平均）
    ↓
Step 6: 收集系统指标（显存、GPU 利用率）
    ↓
Step 7: 输出对比表 + 结论
```

### 实操流程

日常测评按以下步骤执行：

|  步骤 | 操作                                                                          |
| :-: | :-------------------------------------------------------------------------- |
|  ①  | **判断模型类型** — 看 app.py import（`AutoModelForCausalLM` → 对话模型；其他 → 查非对话模型表）    |
|  ②  | **准备测试脚本** — 对话模型：复制 `benchmark_llm.py` 到项目 `code/` 目录。非对话模型：参考对应类型表仿写      |
|  ③  | **停 app.py** — `Ctrl + C` 停 Gradio 进程，释放显存                                  |
|  ④  | **运行测试** — `python benchmark_llm.py`，等待输出完成                                 |
|  ⑤  | **下载报告** — 下载 `{模型名}_benchmark_report.html` + `{模型名}_benchmark_result.json` |
|  ⑥  | **多算力对比（可选）** — 同模型在不同算力各跑一次，用 `benchmark_compare.py` 合并                    |
|  ⑦  | **并发压力测试（可选）** — 部署后 `benchmark_concurrency.py` 测并发承压                       |

### 快速评测脚本

```python
import torch
import time

def benchmark_model(model, tokenizer, prompts, num_runs=10, max_new_tokens=256):
    """
    快速评测（单卡，无并发）
    返回 TPS、峰值显存、平均延迟
    """
    # 预热
    warmup(model, tokenizer)

    # 重置显存统计
    torch.cuda.reset_peak_memory_stats()
    torch.cuda.synchronize()

    all_latencies = []
    total_tokens = 0

    for _ in range(num_runs):
        for prompt in prompts:
            inputs = tokenizer(prompt, return_tensors='pt').to(model.device)
            input_len = inputs['input_ids'].shape[-1]

            # 精确测量推理耗时
            start = time.perf_counter()

            with torch.no_grad():
                outputs = model.generate(
                    **inputs,
                    max_new_tokens=max_new_tokens,
                    do_sample=False,       # greedy 保证可复现
                    pad_token_id=tokenizer.eos_token_id
                )

            torch.cuda.synchronize()
            elapsed = time.perf_counter() - start

            output_len = outputs.shape[-1] - input_len
            total_tokens += output_len
            all_latencies.append(elapsed)

    peak_mem = torch.cuda.max_memory_allocated() / 1024**3
    total_time = sum(all_latencies)
    avg_latency = (total_time / (num_runs * len(prompts))) * 1000

    return {
        'tps': total_tokens / total_time,
        'peak_memory_gb': peak_mem,
        'avg_latency_ms': avg_latency,
        'p50_latency_ms': sorted(all_latencies)[len(all_latencies)//2] * 1000,
        'p95_latency_ms': sorted(all_latencies)[int(len(all_latencies)*0.95)] * 1000,
    }

# 使用示例
# prompts = ["简单问题"] * 10  # 固定 prompt 对比
# result = benchmark_model(model, tokenizer, prompts, num_runs=10)
```

### 两阶段测量法（benchmark\_llm.py 核心设计）

benchmark 脚本通过两次独立 `generate()` 调用来分离 TTFT 和吞吐的测量：

```python
# 第1步：测 TTFT（只生成1个token）
t0 = time.time()
with torch.no_grad():
    first_output = model.generate(input_ids, max_new_tokens=1, ...)
ttft = time.time() - t0

# 第2步：测完整生成（max_new_tokens=512）
t0 = time.time()
with torch.no_grad():
    full_output = model.generate(input_ids, max_new_tokens=512, ...)
e2e = time.time() - t0
throughput = output_count / e2e
```

| 方案                              | TTFT 准确性 | 吞吐准确性 | 说明                        |
| :------------------------------ | :------: | :---: | :------------------------ |
| 一次 generate + 取第一个 token 时间     |   ❌ 不准   |  ✅ 准  | PyTorch 内部可能批处理           |
| 一次 generate(max\_new\_tokens=1) |    ✅ 准   | ❌ 无数据 | 只有 1 个 token 算不出吞吐        |
| **两次独立调用**                      |    ✅ 准   |  ✅ 准  | 实际部署场景每个请求独立，不共享 KV Cache |

## 多推理引擎对比

不同推理引擎在相同模型下的性能差异明显，以下是典型的相对差距（以原生 Transformers 为基准）：

| 引擎              |   TPS 提升  | 显存节省 | 适用场景         |
| :-------------- | :-------: | :--: | :----------- |
| Transformers 原生 | 基准 (1.0x) |  基准  | 通用，配置最少      |
| vLLM（单卡）        |    2-4x   | ∼20% | 高并发生产        |
| TGI             |   1.5-3x  | ∼15% | HF 生态用户      |
| TensorRT-LLM    |    3-6x   | ∼30% | 极致优化，H100 最佳 |

**评测时建议**：如果做量化对比，统一用同一引擎；如果做引擎对比，统一用同一精度。

## 输出质量评估（LLM-as-Judge）

把原模型和优化后模型的输出发给一个更强的模型做盲评。

**推荐做法**：

- 从测试集随机抽 **100 条**
- 全部用裁判模型打分
- 统计"优化后赢了 / 平了 / 输了"的比例

**评估 prompt 模板**：

```
你是一个公正的评估者。给定问题和两个回答，判断哪个更好。
输出格式：{"better": "A 或 B", "reason": "..."}

问题：{prompt}
回答A：{output_before}
回答B：{output_after}
```

**裁判模型不是必须用 GPT-4**。一个能力足够的本地模型（如某 35B 模型）对 ≤14B 的小模型评估完全够用。

## 模型类型适配

不是所有模型都是对话模型。实际项目中已遇到多种模型类型，部署与评测方式各异：

|  #  | 类型         | 判断依据（看 app.py import）                              | 核心指标              |
| :-: | :--------- | :------------------------------------------------- | :---------------- |
|  1  | **对话/LLM** | `AutoModelForCausalLM` + `apply_chat_template`     | TTFT / 吞吐 / E2E   |
|  2  | **推理/CoT** | `AutoModelForCausalLM` + thinking/reasoning 模式     | 同对话 + Thinking 耗时 |
|  3  | **重排序**    | `AutoModelForSequenceClassification` + query+doc 对 | 单 pair 延迟、QPS     |
|  4  | **文本嵌入**   | `AutoModel` + `pooler_output`                      | 编码延迟、QPS          |
|  5  | **翻译**     | `EncoderDecoderModel` / `Seq2Seq`                  | 延迟、吞吐             |
|  6  | **语音合成**   | 输出 wav + TTS 库                                     | RTF（实时率）、首音延迟     |
|  7  | **视觉定位**   | 输入图像 + CV/vision 库                                 | 延迟、FPS、成功率        |
|  8  | **时序预测**   | 非 transformers 自定义类                                | 延迟                |
|  9  | **未知类型**   | 以上都不匹配                                             | 延迟（最低要求）          |

> 因为需要部署多种类多个模型，可以编写一个自进化的模型规范测试脚本，遇到需要测评的新模型可以基于之前的测试脚本进行修改，思路类似前篇的通用镜像。

## 常见问题与真实踩坑

> 其中 tokenizer 返回差异、`eos_token_id` 为列表、FP8 反量化等概念原理，统一见 [00-前置知识](00-前置知识.md) §七，本节只列场景化修复。

| 问题                                           | 原因                                         | 解决方案                                       |
| :------------------------------------------- | :----------------------------------------- | :----------------------------------------- |
| `ModuleNotFoundError: No module named 'app'` | benchmark.py 和 app.py 不在同一目录               | 放对位置到 `code/` 目录                           |
| `CUDA out of memory`                         | app.py 未完全退出                               | `ps aux \| grep python` 杀掉残留进程             |
| 第一次 TTFT 比后面高 2-3 倍                          | 冷启动（GPU kernel 编译）                         | 预热机制已内置处理                                  |
| MiniCPM tokenizer 返回 tensor 而非 BatchEncoding | `return_tensors="pt"` 时直接返回 `torch.Tensor` | 加 `hasattr(tokenized, "input_ids")` 判断分支   |
| `eos_token_id` 需传 list 而非 int                | 部分模型配置为 list 格式                            | `eos_token_id=tokenizer.eos_token_id` 兼容处理 |
| bge-reranker float16 输出 NaN                  | Cross-Encoder logit 范围极大，float16 溢出        | 用 float32，仅需 \~4.4GB 显存                    |
| `total_mem` 属性报错                             | 正确属性名是 `total_memory`                      | 用 `total_memory`                           |
| FP8 在 < 8.9 compute capability 上反量化异常        | FP8 需要 H100+ 硬件支持                          | 低于 H100 用 BF16                             |
| T2I 模型评测维度不完整                                | 文生图需要多维度                                   | ①基础延迟 ②步数影响 ③显存峰值                          |

```python
# 修复示例
tokenized = tokenizer.apply_chat_template(
    messages, add_generation_prompt=True,
    tokenize=True, return_tensors="pt",
)
if hasattr(tokenized, "input_ids"):
    input_ids = tokenized.input_ids
else:
    input_ids = tokenized  # MiniCPM 直接返回 tensor
```

| 现象             | 原因 | 解决                                   |
| :------------- | :- | :----------------------------------- |
| tok/s 看不懂      | —  | 1 token ≈ 1.5 汉字，100 tok/s ≈ 150 字/秒 |
| 不确定 MoE 模型显存需求 | —  | 看激活参数不是总参数。如 `122B-A10B` 按 10B 判断    |

## 注意事项

1. **控制变量原则**：一次只变一个变量。对比量化时框架要一样，对比引擎时精度要一样
2. **预热**：正式测试前先跑 10 次，消除冷启动影响
3. **统计口径**：报 P50 / P95 / P99 而不是只有均值
4. **梯度无关**：评测时全程 `with torch.no_grad()`，关闭梯度计算
5. **`do_sample=False`**：用 greedy decoding 保证可复现结果
6. **`max_new_tokens`** **一致**：不是 `max_length`
7. **显存测量前 reset**：`torch.cuda.reset_peak_memory_stats()`
8. **环境变量一致**：`CUDA_VISIBLE_DEVICES`、`TORCH_DTYPE` 要记录
9. **以 dev run 确认稳定性**：先跑 1 轮确认脚本不崩溃，再跑完整 5-10 轮
10. **MoE 模型注意**：`device_map='auto'` 会自动处理跨卡分配，显存按激活参数估算
