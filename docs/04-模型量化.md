# 04 · 模型量化

运行环境确定之后，下一步是在显存占用与推理质量之间做权衡——这就是量化。精度概念见 [00-前置知识](00-前置知识.md) 的「三、核心术语卡」。

## 精度格式速查

不同精度类型占用的显存差异显著，选择时需综合考虑模型大小、硬件能力和质量要求：

| 精度 | 每参数大小 | 7B 模型 | 14B 模型 | 70B 模型 | 说明 |
|:----|:---------:|:------:|:--------:|:--------:|:----|
| FP32 | 4 字节 | ~28 GB | ~56 GB | ~280 GB | 训练默认，推理极少用 |
| BF16 | 2 字节 | ~14 GB | ~28 GB | ~140 GB | 推理基线，质量最高 |
| FP8 | 1 字节 | ~7 GB | ~14 GB | ~70 GB | 算力≥8.9 才原生支持 |
| INT8 | 1 字节 | ~7 GB | ~14 GB | ~70 GB | 质量损失<1%，通用性好 |
| INT4 | 0.5 字节 | ~3.5 GB | ~7 GB | ~35 GB | 质量损失约 2-5%，性价比最高 |
| 1.58-bit | 0.2 字节 | ~1.4 GB | ~2.8 GB | ~14 GB | 仅特定架构支持（如 BitCPM） |

> 以上为模型权重本身的理论大小，实际加载还需加约 20% 开销（KV Cache + 激活值）。

## bitsandbytes 量化（最易上手）

bitsandbytes 与其他量化方案的核心区别：**不需要预量化过程，在模型加载时即时应用量化**。适合快速验证和资源受限场景。

### NF4 格式（推荐 INT4）

NF4（NormalFloat 4-bit）是一种**信息论最优**的 4-bit 数据类型，专门为正态分布的权重设计。

```python
from transformers import BitsAndBytesConfig, AutoModelForCausalLM
import torch

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",          # NF4 格式
    bnb_4bit_compute_dtype=torch.bfloat16,  # 计算时反量化到 bf16
    bnb_4bit_use_double_quant=True,     # 双重量化
)

model = AutoModelForCausalLM.from_pretrained(
    'model-path',
    quantization_config=bnb_config,
    device_map='auto',
)
```

**双重量化（Double Quantization）** ：对量化参数（scale）本身再做一次 INT8 量化，额外节省约 0.4 bit/参数。对 70B 模型可再省约 3.5GB。

### INT8 加载

```python
model = AutoModelForCausalLM.from_pretrained(
    'model-path',
    load_in_8bit=True,
    device_map='auto'
)
```

> 需要安装 `bitsandbytes`：`pip install bitsandbytes`

### bitsandbytes 适用场景

| 场景 | 推荐 | 理由 |
|:----|:----|:-----|
| 快速验证 | INT4 NF4 | 一行代码加载，无需预量化 |
| QLoRA 微调 | INT4 NF4 + 双重量化 | 低至 24GB 训 70B |
| 生产环境高吞吐 | ❌ 不推荐 | 推理时解量化有额外开销，首次推理慢 |

## 离线预量化：GPTQ

GPTQ 使用少量校准数据（calibration data），通过 Hessian 矩阵优化找到最优量化参数，量化后的模型直接保存为 INT4 权重，推理时无需解量化步骤。

### 原理要点

```
逐层处理：
1. 采集校准数据（128-256 条文本）
2. 对该层输入计算 Hessian 逆矩阵
3. 按权重重要性排序，贪心量化（优先保护重要权重）
4. 每量化一个权重，补偿剩余未量化权重的误差
```

### 使用已量化的模型（最快路径）

HuggingFace 上已有大量社区预量化版本（TheBloke 等）：

```python
from transformers import AutoModelForCausalLM

# 预量化模型通常有 GPTQ 后缀
model = AutoModelForCausalLM.from_pretrained(
    "TheBloke/Llama-2-7B-GPTQ",
    device_map="auto",
    revision="main"
)
```

### 自行量化（需要 auto-gptq）

```bash
pip install auto-gptq optimum
```

```python
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig
from datasets import load_dataset

quantize_config = BaseQuantizeConfig(
    bits=4,              # 4-bit
    group_size=128,      # 每 128 个权重为一组共享量化参数
    desc_act=False,      # 是否按激活值排序（True 精度更高但速度略慢）
)

# 准备校准数据
dataset = load_dataset("wikitext", "wikitext-2-raw-v1", split="train")
calibration = dataset.select(range(128))["text"]

model = AutoGPTQForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B-Instruct",
    quantize_config,
)

model.quantize(calibration)  # 运行量化（~15-30 分钟）
model.save_pretrained("./llama-8b-gptq")
```

## 离线预量化：AWQ

AWQ 的洞察是：约 1% 的"显著性权重"（激活值大的通道）对模型输出影响巨大。

### 原理

```
三步：
1. 统计校准数据上的激活幅度，识别 top 1% 显著性通道
2. 对显著性权重施加缩放因子（量化时保留更多信息）
3. 剩余 99% 正常量化
```

### 使用

```bash
pip install autoawq
```

```python
from awq import AutoAWQForCausalLM

model = AutoAWQForCausalLM.from_pretrained("meta-llama/Llama-3.1-8B-Instruct")
model.quantize(
    tokenizer,
    quant_config={"bits": 4, "group_size": 128}
)
model.save_pretrained("./llama-8b-awq")
```

### GPTQ vs AWQ 对比

| 维度 | GPTQ | AWQ |
|:----|:----|:----|
| 校准数据 | 需要 | 不需要 |
| 量化时间 | 15-30 min | 约 10 min |
| 精度 | 优秀 | 略优（尤其小模型） |
| 推理速度 | 快（含 Marlin） | 快（含 Marlin） |
| 工具链成熟度 | 最成熟 | 成熟 |

> **选型建议**：生产环境优先选 GPTQ（社区版本最多），质量敏感场景选 AWQ。

## FP8 与反量化

FP8 格式的模型在算力 < 8.9 的 GPU 上**自动反量化到 BF16**，显存占用量翻倍。

| GPU | 算力 | FP8 支持 |
|:----|:---:|:--------:|
| H100 | 9.0 | ✅ 原生 FP8 |
| A100 | 8.0 | ❌ 反量化到 BF16 |
| A6000 | 8.8 | ❌ 反量化到 BF16 |
| L40S | 8.9 | ✅ FP8 |

> **算力（Compute Capability）是什么**：NVIDIA 用算力版本号标识 GPU 架构能力，例如 Ampere 的 A100 为 8.0、Ada 的 L40S 为 8.9、Hopper 的 H100 为 9.0。FP8 数据格式需要算力 ≥ 8.9 的硬件才有原生支持；低于此规格的 GPU 在加载 FP8 权重时会自动反量化为 BF16，导致显存占用翻倍。选模型与精度前，先用 `nvidia-smi -q` 或 `torch.cuda.get_device_capability()` 确认显卡算力。

> 🔧 **真实踩坑**：某 35B 模型在某 40GB 单卡规格（算力 8.0）上加载 FP8 权重反量化到 BF16，35B 模型从 ~17.5GB（FP8）膨胀到 ~35GB（BF16），超过 40GB 显存限制导致 OOM。最终启用 CPU offload 才完成推理，速度降至 10 tok/s。**选模型前先查 GPU 算力，不支持 FP8 的机器直接下 BF16 版本更省事**。

## Marlin 内核加速

Marlin 是专为 4-bit 量化模型设计的 GPU 推理内核，可将 GPTQ/AWQ 的推理吞吐提升 30-60%。

| 方案 | 吞吐（tok/s） | 相对 FP16 |
|:----|:-----------:|:--------:|
| FP16 (基线) | ~350 | 1x |
| GPTQ 4bit | ~520 | 1.5x |
| GPTQ + Marlin | ~712 | 2.0x |
| AWQ 4bit | ~550 | 1.6x |
| AWQ + Marlin | ~741 | 2.1x |

> 数据来源：vLLM / Hugging Face 官方 benchmark（Llama-3.1-8B, A100 80GB）

Marlin 在 vLLM 和文本生成推理库中已默认启用，使用预量化 GPTQ/AWQ 模型时自动生效，无需额外配置。

## 量化选型决策树

```
你的场景是？
├─ 快速验证 / 一次性推理
│   └─ bitsandbytes INT4（一行代码，无需预量化）
├─ QLoRA 微调
│   └─ bitsandbytes INT4 NF4 + 双重量化
├─ 生产环境高吞吐
│   ├─ 社区有预量化版本 → GPTQ（最成熟）
│   ├─ 质量敏感 → AWQ（精度略优）
│   └─ 需自行量化 → GPTQ（工具链最完善）
├─ CPU / 边缘设备
│   └─ GGUF（单文件部署，CPU 推理原生支持）
└─ 质量>显存，不舍得量化
    └─ BF16（质量最高，显存最贵）
```

## 量化影响评估

每次量化必须测量三件事：

### 1. 显存

```python
import torch
torch.cuda.reset_peak_memory_stats()
# 做推理
peak_mem = torch.cuda.max_memory_allocated() / 1024**3
print(f'峰值显存: {peak_mem:.1f} GB')

# 还可用 nvidia-smi 交叉验证
# nvidia-smi --query-gpu=memory.used --format=csv -l 1
```

> 🔧 bge-reranker-v2-m3 用 float16 推理时，Cross-Encoder 的原始 logit 范围极大（正样本 >10），float16 最大可表示 65504，高相关度对容易溢出为 inf → 后续运算得 NaN。560M 小模型 float32 仅需 ~4.4GB 显存，完全够用。**小模型不一定需要低精度量化，float32 可能更稳定**。

### 2. 推理速度

```python
import time
start = time.time()
for _ in range(10):
    _ = model.generate(...)
elapsed = time.time() - start
tps = total_tokens / elapsed
print(f'TPS: {tps:.1f}')

# 区分 TTFT（首 token 延迟）和 ITL（后续 token 速率）
# bitsandbytes 首次推理很慢（解量化），GPTQ/AWQ 首次就快
```

### 3. 输出质量

用更强模型做自动评估：把量化前后的输出发给裁判模型盲评。建议测试集 ≥ 100 条。

## 校准数据要求

GPTQ 等离线量化方案需要校准数据（calibration data）来计算最优量化参数。

| 要求 | 说明 |
|:----|:-----|
| 数量 | 128-512 条文本即可 |
| 来源 | wikitext、c4、或目标领域数据 |
| 多样性 | 覆盖模型的使用场景 |
| 格式 | 纯文本，不需要标签 |

```python
from datasets import load_dataset

# wikitext（通用，最常用）
calib = load_dataset("wikitext", "wikitext-2-raw-v1", split="train").select(range(128))

# 目标领域数据（效果更好）
calib = load_dataset("your-domain/data", split="train").select(range(128))["text"]
```

> ⚠️ 校准数据不需要大量，但需要多样。只用 128 条同一主题的文本可能导致量化偏好。

## 显存估算公式

选择量化方案前快速预估显存：

```
模型权重 ≈ 参数量 × 每参数字节数 × 1.2（KV Cache 和激活值的粗略系数）

示例：
- 7B INT4：7e9 × 0.5 × 1.2 ≈ 4.2 GB
- 14B INT4：14e9 × 0.5 × 1.2 ≈ 8.4 GB
- 7B BF16：7e9 × 2 × 1.2 ≈ 16.8 GB
- 14B BF16：14e9 × 2 × 1.2 ≈ 33.6 GB
```

> 🔧 开发环境验证和推理服务部署可以用不同规格。验证阶段显存不够也能跑（CPU offload），重点是确认依赖和代码正确；部署时再加显存追求流畅体验。

## 常见问题

| 问题 | 原因 | 方案 |
|:----|:----|:-----|
| `bitsandbytes` 找不到 | CUDA 版本不匹配 | `pip install bitsandbytes --upgrade` |
| 质量下降 > 10% | 模型太小 / 量化位宽太低 | 用 INT8 或 AWQ 替代 |
| 第一次推理很慢 | bitsandbytes 在推理时解量化 | 换 GPTQ/AWQ 预量化模型 |
| `load_in_4bit` 不支持 | bitsandbytes 版本太旧 | 升级到 0.43+ |
| FP8 模型显存翻倍 | GPU 算力 < 8.9 触发反量化 | 直接下 BF16 版本 |
| 量化后输出 NaN | 某些层对量化特别敏感 | 排除指定层不量化 |
| Cross-Encoder float16 输出 NaN | 输出 logit 范围超出 FP16 上限 | 改用 float32 |

## 量化方法对比

| 方法 | 优点 | 缺点 | 适用场景 |
|:----|:----|:----|:--------|
| bitsandbytes INT4 | 一行代码，无需预量化 | 推理时解量化有开销 | 快速验证、QLoRA |
| bitsandbytes INT8 | 质量损失小 | 显存节省不如 INT4 | 对质量敏感 |
| GPTQ | 预量化，推理更快，生态最成熟 | 需要校准数据 | 预量化社区版本多 |
| AWQ | 质量保持好，无需校准数据 | 工具链略复杂 | 质量优先场景 |
| GGUF | CPU 原生支持，单文件部署 | GPU 推理不如 GPTQ 快 | 边缘/CPU 设备 |
