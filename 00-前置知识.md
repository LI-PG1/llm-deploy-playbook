# 00 · 前置知识

> 在正式开始前你应该知道的一些概念和经验。如果你已经有一定部署经验，可以直接跳到 [01-模型选取与下载](docs/01-模型选取与下载.md)。

## 一、部署是什么

把训练好的模型"跑起来并对外提供服务"。整个过程包含：

```
选模型 → 确认依赖 → 写 Dockerfile → 量化 → 部署 → 加界面 → 评测
```

本手册的 01–09 依次对应上述流程。

## 二、核心术语卡

| 术语 | 解释 |
|:----|:-----|
| **HF Hub** | HuggingFace 模型托管平台。下载模型的地方，有时需要 token（门控模型） |
| **transformers** | HuggingFace 的 Python 库，加载/使用模型的核心工具 |
| **config.json** | 模型架构的配置文件，告诉加载器"这是个什么模型" |
| **safetensors** | 安全的权重文件格式（无代码执行风险），优先使用 |
| **AutoModelForCausalLM** | transformers 中加载对话模型的自动类 |
| **device_map** | 控制模型层放在哪张 GPU/CPU 上，跨卡必备 |
| **trust_remote_code** | 是否允许执行模型目录中的自定义 Python 代码 |
| **vLLM** | 高吞吐推理引擎，适合生产环境 |
| **PagedAttention** | vLLM 的核心技术，像 OS 管理内存一样管理 KV Cache |
| **Gradio** | 为模型快速搭建 Web 界面的 Python 库 |
| **Marlin** | 专为 4-bit 量化模型设计的 GPU 推理加速内核 |
| **P50/P95/P99** | 百分位延迟指标。P50 是中位数，P95 是"95% 的请求比这个快" |

## 三、模型来源

| 来源 | 特点 |
|:----|:-----|
| **HuggingFace Hub** | 最全，社区版本最多。国内可走 hf-mirror 镜像 |
| **ModelScope** | 国内访问快，部分模型首发在此 |
| **官方 GitHub** | 部分模型只开源代码，权重放 HF |

> **门控模型**：在 HF 模型页有 `→Use this model` → `Access granted` 按钮的模型。必须用 HF Token 下载。镜像站点不带 token 下载门控模型时，文件大小正常但权重是随机占位数据，模型加载不报错但输出乱码。

## 四、Docker / GPU 基础

### 显存估算

```
模型权重 ≈ 参数量 × 每参数字节数 × 1.2（KV Cache + 激活值）

BF16: 7B → ~16.8GB,   14B → ~33.6GB
INT4: 7B → ~4.2GB,    14B → ~8.4GB
```

### GPU 算力与 FP8

| GPU | 算力 | FP8 支持 |
|:----|:---:|:--------:|
| H100 | 9.0 | ✅ 原生 |
| L40S | 8.9 | ✅ 原生 |
| A100 | 8.0 | ❌ 反量化到 BF16 |
| A6000 | 8.8 | ❌ 反量化到 BF16 |

> FP8 模型在算力 < 8.9 的 GPU 上自动反量化到 BF16，**显存翻倍**。如果用了 FP8 单卡部署 OOM，先确认是不是反量化导致的。

### 卡间通信策略

| 策略 | 原理 | 显存节省 | 通信开销 |
|:----|:----|:-------:|:-------:|
| Tensor Parallel (TP) | 每层切分到多卡 | ~1/N | 高（每层 AllReduce） |
| Pipeline Parallel (PP) | 不同层分不同卡 | ~1/N | 低（仅层边界） |
| Data Parallel (DP) | 每卡完整模型 | 不省 | 低（仅梯度同步） |
| Expert Parallel (EP) | 不同专家不同卡 | ~1/N | 中（All-to-All） |

## 五、主线总览

从选模型到对外提供服务的完整流程：

```
选模型（来源、规格、可行性）
    ↓
确认依赖与版本（transformers / tokenizers / CUDA 兼容 + 模型-版本匹配）
    ↓
编写 Dockerfile（选基础镜像、瘦身、安全、缓存）
    ↓
模型量化（选量化方案：BNB / GPTQ / AWQ / FP8）
    ↓
vLLM 部署（高吞吐、并发、多卡）
    ↓
Gradio 界面（聊天、Blocks、多模态、后台加载、保活）
    ↓
MoE 模型（专家并行、device_map、EPLB、量化特殊考虑）
    ↓
代码适配（路径、设备、tokenizer、generate 参数、LoRA）
    ↓
模型评测（TTFT/吞吐/P50-P95、预热、并发、对比）
```

## 七、跨篇原理

以下概念在各篇中多次用到，统一归纳在此。

### FP8 反量化

FP8 格式的模型在算力 < 8.9 的 GPU 上**自动反量化到 BF16**，显存占用翻倍。

- H100 (CC 9.0) / L40S (CC 8.9) → 原生 FP8，1 字节/参数
- A100 (CC 8.0) / A6000 (CC 8.8) → 反量化到 BF16，2 字节/参数，显存翻倍
- 选模型前先查 GPU 算力，不支持 FP8 的机器直接下 BF16 版本更省事

### eos_token_id 可能是列表

部分模型（如 Qwen 系列）的 `config.json` 中 `eos_token_id` 是列表格式：

```json
"eos_token_id": [2, 24]
```

如果在 `generate()` 中只传 `eos_token_id=eos_token_id[0]`，模型可能在第一个停止条件后继续生成直到 max_new_tokens。**正确做法**：

```python
model.generate(..., eos_token_id=model.config.eos_token_id)
```

### tokenizer 返回对象的差异

不同模型调用 `apply_chat_template(return_tensors="pt")` 时的返回类型不同：

| 返回类型 | 典型模型 | 处理方法 |
|:--------|:--------|:--------|
| `BatchEncoding`（有 `.input_ids` 属性） | 大多数 transformers 模型 | `input_ids = tokenized.input_ids` |
| 直接返回 `torch.Tensor` | MiniCPM 系列 | `if hasattr(tokenized, "input_ids"):` 分支判断 |

```python
# 兼容写法
tokenized = tokenizer.apply_chat_template(messages, ..., return_tensors="pt")
if hasattr(tokenized, 'input_ids'):
    input_ids = tokenized.input_ids
else:
    input_ids = tokenized  # 直接就是 tensor
```

### transformers 5.x 变化

| 变化 | 影响 | 方案 |
|:----|:----|:-----|
| `apply_chat_template` 返回 `BatchEncoding` | 代码中 `shape` 调用报错 | `tokenized["input_ids"]` 取值 |
| `max_length` 弃用 | `generate(max_length=...)` 报错 | 改为 `max_new_tokens` |
| `all_special_tokens_extended` 移除 | vLLM + Qwen2 报错 | import vLLM 前加 monkey-patch |
| 新架构模型支持 | `qwen3_5_moe` 等需 5.x | 不低于 5.6 |

### Local files only 与绝对路径

`huggingface_hub` 1.x 的 `validate_repo_id` 不认绝对路径。解决：

```python
# app.py 首行
from huggingface_hub import validate_repo_id
validate_repo_id = lambda x, **kwargs: None
```

同时设置环境变量 `TRANSFORMERS_OFFLINE=1` 或 `HF_HUB_OFFLINE=1`。

### disk_multiplier 多卡多进程

部署平台多 GPU 场景需要 `torchrun` 而非 `python app.py` 启动：

```bash
# 单节点多卡（RPC 模式）
torchrun --nproc_per_node=1 --nnodes=1 app.py

# 如果 device_map='auto' 已自动用满多卡，无需 torchrun
```

### trust_remote_code 判断规则

| 情况 | 设？ |
|:----|:----|
| `config.json` 的 `architectures` 是标准类名（`Phi3ForCausalLM`, `LlamaForCausalLM`, `Qwen2ForCausalLM`） | ❌ 不设，用 transformers 内置实现 |
| 自定义架构（`MiniCPMForCausalLM`, `Qwen3ForCausalLM`、Vision-Language Projector） | ✅ 设 `trust_remote_code=True` |
| 模型发布时间早于 transformers 最新稳定版 | ✅ 设 |
